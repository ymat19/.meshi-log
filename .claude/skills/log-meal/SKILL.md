# 食事記録 (log-meal)

ユーザーが食事の写真を送る／記録を依頼したら、以下の手順で記録する。
推定値であることを必ず明示する。

### 最重要：コミット前に必ず1回確認する。ただし1ターンにまとめる（過去の失敗から）

毎日やる作業なので**ユーザーにターンを返す回数を最小化する**。だが推定値を**勝手に
コミットしない**——コミット前に必ず1回、内容を見せて確認・訂正を受ける。鉄則：

- **自分で解けることは聞かない。ただし食事の時刻・食数が写真から確定できないときは
  必ず聞く（下の「絶対厳守（2026-07-09）」参照）。** 料理の場面が写っていて時間帯が
  明白なとき（「炊き込みご飯＋味噌汁＝朝、焼きそば＝昼、牛丼＋酒＝夜」）**だけ**
  自動で食事単位に分割してよい。パッケージ・飲み物だけの写真は時刻を語らない——
  **現在時刻を食事時刻と決めつけてはならない。** 1枚＝1食とも4枚＝1食とも限らない。
  **勝手に全部を1食にまとめない。** 既製品の成分は裏で調べる。
- **確認は1ターンにまとめる。** 推定した全内訳（食事の分割・料理ごとの成分・写真
  ごとの対応・既製品の出典）を**1回でまとめて提示し、訂正と確認を受ける**。本当に
  不明な点があれば、その質問も**同じ1ターンに同梱**する（`AskUserQuestion`）。
- **小出し・後出し・二重確認をしない。** 質問を複数ターンに割らない。提示した後に
  もう一度「確定していい？」と聞き直さない（最初の提示が確認そのもの）。
- 順序は **提示・確認 → 訂正反映 → コミット & push**。確認なしのコミットも、確認の
  ための無駄な往復も、どちらも禁止。

### 絶対厳守（2026-07-03 の重大失敗から。破ると記録が壊れる）

過去に、これらを破って**銀杏をピスタチオと誤認**し、**ユーザーの「豆腐」を勝手に
「たまご豆腐」に改ざん**し、**「全部食べた」と言われた量を二度聞く**という最悪の
体験を招いた。二度と繰り返さない。以下は例外なし。

1. **品目を命名・推定する前に、まず `public/data/*.json` を grep して過去ログを見る。**
   過去ログ確認は「検索で見つからない時の代替」ではなく、**あらゆる品目の同定・
   数値決めの最初の一歩**。写真の品を見た目だけで当てない。過去に自分が記録した品
   （銀杏・かまぼこ・豆腐・炊き込みご飯・ストロング等）は、まず grep で拾って
   名前・数値・容量をそのまま流用する。**過去ログを見れば防げた誤認は「無能」と
   同義。推測の前に必ず grep。**
2. **ユーザーが答えた語は一字一句そのまま記録する。勝手に具体化・変換・「格上げ」
   しない。** 「豆腐」と言われたら item 名は「豆腐（絹豆腐など過去ログの実値）」で
   あって、色や見た目を根拠に「たまご豆腐」等へ書き換えるのは**ユーザー回答の改ざん**
   で厳禁。**一次情報（本人の発話）＞ 自分の推測**。語を変える必要があるなら、記録
   せずに `AskUserQuestion` で聞く。黙って別物に置き換えない。
3. **ユーザーが既に答えたことを二度聞かない。** 「全部食べた」＝量は「写真に写って
   いる全量」。本人が述べた事実（品名・量・区分・銘柄）は確定情報として即採用し、
   質問に化けさせない。分かっていることを聞くのは無駄なターンで、体験を最悪にする。
4. **確認の `AskUserQuestion` を出す前に、過去ログ検索を必ず済ませる。** 選択肢は
   過去ログに基づく具体値にする。画像はファイル名（IMG_xxxx）で参照せず、**写真の
   中身（「お酒の缶・かまぼこが写ってる写真」等）で説明する**。ユーザーにファイル名は
   見えない。

### 絶対厳守（2026-07-09 の重大失敗から。破ると記録が全滅する）

過去に、**朝食と昼食を全部「夕食」に丸め**、**ユーザーが承認する前に推定のまま本番へ
push して誤データを公開→削除**し、**「メダルモッツァレラ」を「とろけるスライスチーズ」に
すり替える**という最悪の体験を招いた。原因は「現在時刻＝食事時刻」「自分の提示＝承認」
という思い込み。二度と繰り返さない。以下は例外なし。

5. **現在時刻を食事時刻と決めつけない。食事の時刻・食数は勝手に確定しない。**
   食事写真は後からまとめて送られることが多く、**「今が夜だから全部夕食」は誤り**。
   商品パッケージ・飲み物・寄りの料理など**時間帯が写真から読めない**場合、
   および**複数の食事にまたがり得る**場合は、**必ず `AskUserQuestion` で「いつ・何食に
   分かれるか」を聞く**。特に**全部を1食（現在時間帯の区分）にまとめるのは禁止**。
   - 例外は、料理の場面＋文脈で時間帯が明白なとき（朝の食卓・夜の晩酌等）だけ。
   - ユーザーが「まだ夕飯食ってない」「これは朝と昼」等と言ったら、その語を最優先で
     採用し、絶対に上書きしない。
6. **ユーザーが食事の分割・時刻・内容を確定承認するまで、`main`（本番）へ push しない。**
   **自分が提示しただけ／`AskUserQuestion` が拒否・未回答のままを「承認」とみなさない。**
   推定のまま本番反映し、あとから公開データを消すのは最悪。順序を厳守：
   **提示・確認 → ユーザーの確定回答 → 訂正反映 → コミット & 本番 push**。
   確認が取れないうちは記録を保留する（勝手に進めない）。
7. **ユーザーが商品の種類・形状を明示したら、その語が指す“まさにその商品”を検索する。**
   「メダル」「フレッシュ」「絹」「6P」等の指定を無視して、名前が似た別カテゴリの商品
   （例：「フレッシュモッツァレラ メダル」≠「モッツァレラとろけるスライス」）で
   代用しない。**検索の第一ヒットに飛びつかず、ユーザーの語に一致する商品か必ず確認**
   してから数値を採る。特定できなければ黙って代用せず `AskUserQuestion` で聞く。

## 1. 推定

写真（と本文）から推定する：

- **items**: 料理名の配列（例: 「醤油ラーメン」「半チャーハン」）
- **type**: `breakfast` / `lunch` / `dinner` / `snack`。**時刻が写真から読めないなら
  現在時刻で決めず、必ず聞く**（絶対厳守 5）。少しでも迷えば聞く。
- **datetime**: JST の ISO8601（`+09:00`）。ユーザーが時刻を述べたらそれに従う。
  **写真に時刻の手がかりが無いのに現在時刻を既定値として確定してはならない**——
  後追い記録では今の時刻＝食事時刻とは限らない（絶対厳守 5）。
- **items[].nutrition**: 料理ごとの栄養成分。**これが唯一の真実で、1食の合計は
  item の合計として自動導出される**（エントリに合計フィールドは無い）。各 item に
  必須8成分を必ず入れる:
  `energy_kcal` `protein_g` `fat_g` `saturated_fat_g` `carbohydrate_g`
  `sugar_g` `fiber_g` `salt_g`（`alcohol_g` は飲んだ item にのみ）。
  - 必須成分が一つでも欠けると `validate-data.ts`／husky pre-commit がコミットを
    止める。**「分からないから 0」は禁止**（0 はその成分を含まない時だけ）。
  - 糖質 `sugar_g` = `carbohydrate_g − fiber_g` で算出する。
- **tags**: 例 `外食` `自炊` `中食` `麺` `揚げ物` `お酒` など

### 栄養値の決め方は優先順を守る（記憶ベースの推定は最終手段）

各 item の栄養値は、必ず次の優先順で決める。**上位で確定できるなら下位に降りない。
記憶での推定は、上の手段がすべて尽きたときの最終手段。**

> **前提（例外なし）：どの品目も、値を決める前にまず `public/data/*.json` を grep して
> 過去ログを確認する。** 過去に記録した品はその実値・容量をそのまま流用する（上の
> 「絶対厳守」1 参照）。過去ログ確認は下記2の一部ではなく、**すべての品目の共通の
> 第一歩**。

1. **既製品はウェブ検索（必須・例外なし）。** コンビニ・スーパー・冷凍食品・
   スナック・飲料・チェーン店メニューなど、**商品名が特定できる既製品は、記憶で
   数値を埋めてはならない。必ず WebSearch / WebFetch で公式・信頼できる情報源を
   引き、パッケージ表示の栄養成分（エネルギー・P・F・C・塩分相当量など）に基づいて
   値を入れる。**
   - 対象の目安: コンビニPB（セブンプレミアム等）、メーカー製品（蒙古タンメン中本、
     6Pチーズ、〇〇ハム等）、缶チューハイ・ビール等の酒類、菓子・スナック、
     外食チェーンのメニュー。
   - 検索クエリは「商品名 + 栄養成分 / カロリー / 成分表示」で引く。**メーカー公式 →
     コンビニ公式 → 信頼できる栄養DB/まとめ**の優先順で採用する。個人ブログの
     又聞き数値だけに頼らない。
   - 商品名は特定できるが容量・枚数等が不明なときは**ユーザーに確認**してから数値を確定。

2. **検索で見つからない／商品特定できないものは、まず過去の記録を探す。** 既製品で
   情報源が出ない場合も、手作り・自炊・盛り合わせの場合も、**いきなり記憶で推定せず、
   先に `public/data/*.json` を検索して同一・類似の品目が過去に記録されていないか
   確認する**（品名で grep するなど）。一致する記録があれば、その値・容量を流用して
   整合させる。**過去の自分の記録と矛盾する推定をしない。**（例：過去に「ミニ
   カマンベール25g」で記録しているのに、今回だけ記憶で「丸ごと90g」と過大推定する、
   は禁止。これは実際に起きた失敗。）

3. **それでも根拠が無いときに限り、記憶ベースの推定を最終手段として使う。** その場合は
   必ず「記憶ベースの推定」とラベルして提示し、ユーザーの判断を仰ぐ（黙って推定値を
   確定値として出さない）。手作り・自炊などで分量を見積もるときも、推定である旨を明示する。

検索で得た**情報源 URL、または流用した過去記録の出典は必ず控え、ステップ2の報告と
ステップ4の記録（item メモ）に残す**。

## 2. コミット前の確認（毎回必須・1ターンにまとめる）

**コミットする前に必ず1回**、次を報告して確認・訂正を受ける。不明点が無くても
**解釈が正しいかを確認するために毎回必ず行う**（省略禁止）。報告内容:

1. **朝/昼/晩（＋間食）への分割と、各食事の内容をどう解釈したか**
   — どの写真の何を、どの食事に入れたか。
2. **料理ごとの内訳** — 各 item の必須8成分（合計はここから自動導出される）。
3. **写真ごとの対応** — どの写真に何が写っているか（見落とし・重複防止）。
4. **既製品の出典 URL** — 行ごとに併記。URL を出せない既製品は「記憶ベース推定」と明示。

**本当に不明な点があれば、この同じタイミングで `AskUserQuestion` を使いまとめて聞く**
（**食事の時刻・食数が写真から確定できない**、量・区分が判別不能、容量不明な既製品、
ユーザーが指定した商品を特定できない 等——絶対厳守 5・7）。自分で解けること（時間帯が
明白な写真の振り分け、自明な区分）だけは聞かない。**時刻の手がかりが無いのに「夕食で
確定」等と決め打ちしない。**

**ユーザーの確定回答が取れるまでコミット・本番 push はしない（絶対厳守 6）。** 提示への
返答が来ない／`AskUserQuestion` が拒否されたら、推定のまま進めず、内容を確定できる形で
もう一度確認する。

**訂正の反映は「ユーザーが言ったとおり」に行う。** 返ってきた回答の語を勝手に別物へ
置き換えない（「豆腐」→「たまご豆腐」等は改ざん・厳禁）。回答済みの品名・量・銘柄を
記録後にもう一度聞き直さない。上の「絶対厳守」2・3 を必ず守る。

**これ以外の往復は一切しない。** 提示後にもう一度「確定していい？」と聞き直さない
（最初の提示が確認そのもの）。小出し・後出し・二重確認は禁止。訂正を反映したら
ステップ3以降の記録に進む。

## 3. 画像保存

画像は **WebP・長辺1000px** に最適化して `public/images/YYYY/MM/` に保存する。
最適化には同梱スクリプト（`sharp` 依存。リポジトリに記録済み）を使う:

```
node scripts/optimize-image.mjs <受け取った画像> public/images/YYYY/MM/DD-HHmm-<type>.webp
```

- `node_modules` が無ければ先に `npm ci` を実行する（リモート環境は使い捨てのため、
  ツールは npm 依存として復元する。apt/mise 等で入れても次セッションには残らない）
- `photos` には公開パス（先頭の `public/` を除いた `images/YYYY/MM/DD-HHmm-<type>.webp`）
  を入れる

## 4. データ追記

確定した MealEntry を一時ファイル（例 `/tmp/entry.json`）に書き、専用スクリプトで
月別 JSON と index に安全に追記する（JSON の手編集はしない）:

```
node scripts/add-meal.mjs /tmp/entry.json
```

MealEntry の形（`src/data/types.ts` が唯一の真実）:

```json
{
  "id": "2026-06-19T12:30:00+09:00-lunch",
  "datetime": "2026-06-19T12:30:00+09:00",
  "type": "lunch",
  "photos": ["images/2026/06/19-1230-lunch.webp"],
  "items": [
    {
      "name": "醤油ラーメン",
      "nutrition": {
        "energy_kcal": 720, "protein_g": 26, "fat_g": 22, "saturated_fat_g": 7,
        "carbohydrate_g": 95, "sugar_g": 90, "fiber_g": 5, "salt_g": 6.8
      }
    }
  ],
  "memo": "",
  "tags": ["外食", "麺"],
  "estimated": true,
  "schemaVersion": 1
}
```

- `id` は `<datetime>-<type>`
- `items[].nutrition` が唯一の真実。ステップ2で報告した料理ごとの値を**そのまま記録する**。
  **1食の合計はエントリに持たせない**（item から自動導出するため。合計フィールドを
  書いても無視され、二重管理になるので書かないこと）
- **既製品の出典 URL は `memo` に必ず残す**（型に出典専用フィールドが無いため）。
  「商品名: URL」の形で、調べた既製品ぶんを列挙する。後から数値の根拠を辿れるようにする。

## 5. コミット & デプロイ

`public/data/` と `public/images/` の変更を、**デプロイ対象ブランチ（既定 `main`）**
にコミットして push する。記録はデータなので PR は不要、直接 push でよい。
`main` への push で Cloudflare Workers Builds が再ビルド→デプロイする（数十秒〜1分）。

- **前提：ステップ2でユーザーの確定承認が取れていること（絶対厳守 6）。** 未確認の
  推定を本番へ出さない。`main` は即公開なので、誤りは全世界に出てから消す羽目になる。

### 誤って記録した／本番に出た記録の訂正・削除

一度 push した記録が誤っていたら（食事区分の取り違え・重複等）、**該当エントリを消して
から**正しく記録し直す。専用の削除スクリプトは無いので、月別 JSON を安全に読み書きする
（手編集はしない・`id` で filter する）：

```
node -e 'const fs=require("fs"),p="public/data/YYYY-MM.json";const d=JSON.parse(fs.readFileSync(p,"utf8"));fs.writeFileSync(p,JSON.stringify(d.filter(e=>e.id!=="消したいID"),null,2)+"\n")'
```

削除後は不要になった画像も `git rm` し、正しいエントリを `add-meal.mjs` で追記 →
`validate-data.ts` → コミット → `main` push → 本番から旧エントリが消え新エントリが
出ることを確認する。
